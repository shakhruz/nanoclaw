# Telegram Emoji Reactions

Двусторонняя система эмоджи-обратной связи между Шахрузом и Милой.

**Зачем:** быстрая обратная связь без текстового ответа. Ты задал длинный вопрос — Мила сразу ставит 👀 и ты понимаешь что она взяла в работу. Мила прислала draft — ты реагируешь 👍 и она выполняет, не надо писать "да".

---

## ⚠️ Ограничение Telegram Bot API

Telegram разрешает боту реагировать только **эмодзи из предустановленного whitelist'а** (~74 стандартных реакции). Custom emoji только у Premium пользователей. Если попытаться среагировать неразрешённым эмодзи — API вернёт `REACTION_INVALID`, реакция не поставится.

Весь emoji-set ниже специально подобран из разрешённых Telegram'ом. Не меняй эмодзи без проверки — иначе silent fail.

---

## Часть 1 — Мила реагирует на сообщения Шахруза (status signals)

Когда Шахруз присылает сообщение — Мила **сразу** ставит `add_reaction` как статус-сигнал. Не нужно ждать ответа, эмоджи приходит мгновенно.

### Статусные эмоджи (Мила → Шахруз)

| Эмоджи | Значение | Когда использовать |
|--------|----------|--------------------|
| 👀 | Вижу, начинаю обрабатывать | Получила сообщение, приступаю |
| ⚡ | Работаю над длинной задачей | Browser, research, ingest, PDF, файлы, Parallel task |
| 👏 | Готово | Задача выполнена успешно (было ✅ — не в whitelist) |
| 🤔 | Нужно уточнение | Недостаточно информации для продолжения (было ❓) |
| 🫡 | Запланировано / принято к исполнению | Создала scheduled task или отправила в очередь (было 🕐) |
| ✍ | Занесла в wiki | Ingest выполнен, страница создана (было 📝) |
| 🙏 | Выполняю повторно | Шахруз попросил повторить, начинаю заново (было 🔁) |
| 💔 | Ошибка / не смогла | Task failed, не хватило прав, API отказал (было ❌) |
| 🙊 | Понял, молчу | На сообщение не буду отвечать текстом — просто принято (было 🤫) |

### Правила для Милы

1. **Реагируй первым, отвечай потом.** На **каждое** входящее сообщение Шахруза — сначала `add_reaction` с `👀`, затем начинай работать.
2. **Обновляй статус реакцией.** После завершения задачи смени `👀` → `👏` (или другой финальный эмоджи). Это делается новым вызовом `add_reaction` на тот же `message_id` — Telegram заменяет реакцию.
3. **Не злоупотребляй.** Реакции — сигналы, не украшения. **Один эмоджи** на сообщение в моменте.
4. **Для scheduled tasks и voice** — сначала `🫡` или `👀`, после выполнения `👏`.
5. **Если эмоджи не поставился** (silent fail из-за REACTION_INVALID в будущем) — отправь короткое текстовое сообщение как fallback и залогируй факт, чтобы можно было обновить whitelist.

### 🔴 CRITICAL: порядок tool-вызовов в turn'е

Claude Agent SDK берёт `result` (финальный текст) *только с последнего assistant-turn'а с stop_reason=end_turn*. Если последний tool_call — `add_reaction` (т.е. turn заканчивается на реакцию), `result` приходит пустой, и текст в Telegram *не отправляется*.

**Правильный порядок каждого turn'а:**
```
1. add_reaction (👀 или ⚡ или 🫡) ← статус в начале
2. Основная работа (любые tools: Bash, Read, Edit, skills, send_message для промежуточных апдейтов)
3. add_reaction (👏 или ✍ или 💔) ← финальный статус
4. [ТЕКСТ] ← обязательно ПОСЛЕДНИМ, это твой ответ Шахрузу
```

**Неправильный порядок (приводит к пропаже текста):**
```
1. add_reaction 👀
2. работа
3. [ТЕКСТ]
4. add_reaction 👏  ← БАГ: text теряется, turn заканчивается на tool_use
```

Если ты шлёшь ответ через `mcp__nanoclaw__send_message` (MCP тул) — это не требует быть последним, т.к. сообщение уходит немедленно в IPC. Но основной текстовый ответ без send_message должен быть *последним* блоком turn'а.

### Шпаргалка: «текст-last»

```
react 👀 → работа → react 👏 → текст ✓
react 👀 → send_message("…") → react 👏 → текст ✓ (send_message уходит сразу, финал-текст тоже)
react 👀 → работа → текст → react 👏 ✗ (последний блок tool_use → текст теряется)
```

### Пример флоу

```
Шахруз: "Сделай дайджест почты"
  → Мила: react 👀 (сразу)
  → Мила: [mcp__gmail__search_messages, формирует дайджест...]
  → Мила: react 👏 (после отправки дайджеста)
  → Мила: [текстовый ответ с дайджестом]
```

```
Шахруз: "Исследуй паттерны утренних рутин через Parallel"
  → Мила: react ⚡ (длинная задача)
  → Мила: [запускает parallel-task, создаёт scheduled polling]
  → Мила: react 🫡 (task создан, ждём результата)
  → Мила: [текст: "Запустила глубокий ресёрч, run_id=X, пришлю результат через ~10 мин"]
  → (через 10 минут)
  → Мила: react 👏 (на исходное сообщение — задача полностью закрыта)
  → Мила: [текст с результатом ресёрча]
```

## Часть 2 — Шахруз реагирует на сообщения Милы (command emojis)

Когда Мила получает маркер `[Reaction: X] on message <id>` — это **команда** от Шахруза на её собственное предыдущее сообщение.

### Командные эмоджи (Шахруз → Мила)

| Эмоджи | Команда | Действие Милы |
|--------|---------|---------------|
| 👍 | Да / Подтвердить / Одобрить | Выполнить последнее pending действие (отправить черновик, создать задачу, и т.д.) |
| 👎 | Нет / Отклонить / Отменить | Отменить последнее pending действие |
| ❤ | Запомни / В wiki | Ingest содержимого сообщения в wiki как источник |
| 🤩 | Повтори | Повторить последнее действие с обновлёнными данными (было 🔁) |
| 🔥 | Важно / Закрепи | Добавить в `wiki/inbox.md` с пометкой важное (было ⭐) |
| 👌 | Закрой задачу | Закрыть последнюю упомянутую задачу в Todoist (было ✅) |
| 🤬 | Стоп / удали | Отменить запущенную задачу или удалить последнее созданное (было ❌) |
| 🤔 | Подробнее | Объясни подробнее про это сообщение (одинаково с Милиным status "нужны данные" — различается направлением) |
| 🫡 | Напомни / follow-up | Добавить в `wiki/inbox.md` как follow-up reminder (было 📌) |
| 😴 | Тихо | Не отвечать на это сообщение текстом (было 🔕) |

### Правила для Милы при получении реакции

1. **Найди контекст.** В сообщении `[Reaction: X] on message <msgId>` — найди в истории чата **твоё собственное** сообщение с этим ID, чтобы понять на что отреагировал Шахруз. Если не можешь найти — отреагируй `🤔` и спроси коротко.
2. **Действуй молча.** Не объясняй «я получила реакцию». Просто сделай что просят. После выполнения — поставь `👏` на то исходное сообщение, на которое Шахруз отреагировал.
3. **Фильтруй свои собственные реакции.** Handler Telegram'а не должен доставлять тебе твои собственные реакции как команды (это делает channel-level фильтр по `user.id`). Если всё-таки прилетела реакция с `sender == bot_id` — игнорируй.
4. **Подтверди успех реакцией.** После выполнения команды — `add_reaction` с `👏` на ИСХОДНОЕ сообщение (то, на которое реагировал Шахруз), не на маркерное событие.
5. **Если непонятно** — реагируй `🤔` и спроси коротко текстом.

### Пример флоу

```
Мила: "Добавить задачу 'Позвонить Ивану' на завтра 14:00 в Inbox, p4? [draft]"
  → Шахруз: реагирует 👍 на сообщение Милы
  → Мила получает: [Reaction: 👍] on message <id-drafta>
  → Мила: (находит свой draft по id → видит что это Todoist create)
  → Мила: mcp__todoist__create_task → успех
  → Мила: add_reaction(<id-drafta>, '👏')
  → (больше ничего; текстовой реплики не надо)
```

```
Мила: "Вот интересная статья Karpathy про LLM Wiki: [URL]"
  → Шахруз: реагирует ❤ на сообщение Милы
  → Мила получает: [Reaction: ❤] on message <id-статьи>
  → Мила: (находит сообщение → URL → запускает ingest pipeline)
  → Мила: curl → source-summary → concepts → index → log → git commit
  → Мила: add_reaction(<id-статьи>, '✍')
  → (после возможен короткий текст: "Готово, затронула 3 страницы")
```

```
Мила: "Создала scheduled task 'Позвонить бухгалтеру' на пятницу 10:00"
  → Шахруз: реагирует 🫡 на сообщение Милы
  → Мила: (понимает: follow-up, Шахруз хочет чтоб она ещё раз напомнила)
  → Мила: добавляет строку в wiki/inbox.md: "[2026-04-12] Не забыть напомнить про звонок бухгалтеру в пт 10:00"
  → Мила: add_reaction(<id-сообщения>, '👏')
```

## Часть 3 — Установка (Code Changes)

**Статус:** ✅ Применено на main (см. commit `feat(telegram): two-way emoji reactions channel`)

Реализация из 5 файлов:

1. **`src/channels/telegram.ts`** — `bot.on('message_reaction')` handler (с фильтром `user.id != bot.id`); публичный метод `reactToMessage(jid, messageId, emoji)` через `bot.api.setMessageReaction`; `allowed_updates: ['message', 'message_reaction']` в `bot.start()`
2. **`src/types.ts`** — опциональный `reactToMessage?()` в интерфейсе `Channel`
3. **`src/ipc.ts`** — обработчик IPC события `type: 'react'` с проверкой авторизации (группа-источник IPC должна совпадать с владельцем chat_jid) и делегацией на channel
4. **`container/agent-runner/src/ipc-mcp-stdio.ts`** — новый MCP-инструмент `add_reaction(message_id, emoji)` — пишет IPC-файл, не делает прямого channel-вызова
5. **`src/index.ts`** — функция `reactToMessage` в `IpcDeps`, которая находит нужный канал через `channels.find(c => c.ownsJid(jid))` и вызывает его метод

### Технические детали

- **Сенкендер bot'а:** при отправке реакции через `setMessageReaction` Telegram **не** доставляет эту же реакцию обратно боту как `message_reaction` update — так что фильтр по `user.id` нужен для теоретической безопасности, в нормальном flow этот кейс не срабатывает
- **`allowed_updates`:** grammy по умолчанию НЕ подписывается на `message_reaction`, его надо явно добавить в конфиг `bot.start()`. Иначе Telegram не будет присылать update-ы
- **Группа должна быть у бота в админке**, чтобы бот видел реакции других пользователей в group chats. В личных DM (как наш `telegram_main`) это не требуется — реакции приходят всегда
- **Тип emoji в API:** `[{type: 'emoji', emoji: 'X'}]` — тип `'emoji'` для стандартных, `'custom_emoji'` для custom (только Premium)

### Troubleshooting

- **Реакция не ставится, silent fail**: проверь что emoji в whitelist'е. Логи контейнера: `grep "Failed to send Telegram reaction" logs/nanoclaw.log` — если видишь `REACTION_INVALID`, значит emoji не разрешён
- **Шахрузовы реакции не приходят**: проверь что бот стартанул с `allowed_updates: ['message', 'message_reaction']` — в логах должно быть это при `Telegram bot connected`. Если нет — бот стартовал без подписки
- **Мила видит свои собственные реакции как команды**: проверь что handler фильтрует `user.id === bot.id` — если фильтр отсутствует, будет self-echo loop
